# HCPCS Code Prediction Pipeline

A hybrid retrieval, hierarchical-funnel, and agentic-search architecture for predicting HCPCS Level II codes from raw `item_name` + `supplier_name` input, designed to push effective accuracy on auto-coded items above 90% by combining a curated master dictionary, confidence-gated automation, and human review for the residual.

## Master dictionary schema

| Column | Description |
|---|---|
| `hcpcs_code` | HCPCS Level II code (primary key) |
| `code_class` | Top-level taxonomy bucket |
| `major_class` | Mid-level category within `code_class` |
| `minor_class` | Fine-grained category within `major_class` — the funnel narrows to this bucket before retrieval runs |
| `long_description` | Full CMS Book 2 long descriptor (carries the disambiguating detail the short descriptor lacks) |
| `embedding` | Vector embedding of `long_description` (text-embedding-3-large) |

## Pipeline stages

**1. Normalize & classify.** A single LLM call (GPT-4.1-mini) cleans the raw item text (expanding domain abbreviations), extracts structured attributes (material, size, laterality, disposable vs. reusable, unit), and predicts a `code_class` → `major_class` → `minor_class` path. The path prediction is assisted by precomputed centroid embeddings (mean of member `long_description` embeddings) at each taxonomy level, giving the LLM a shortlist to confirm rather than a blind classification over hundreds of values.

**2. Supplier cache lookup.** Exact and fuzzy match against a table of previously confirmed `(item_name, supplier_name) → hcpcs_code` pairs. Repeat suppliers resend near-identical descriptions, so this layer resolves a large share of volume at near-perfect accuracy before any retrieval or LLM cost is incurred.

**3. Hierarchical funnel.** On a cache miss, narrow the full code set to the predicted `minor_class` bucket, typically a few dozen codes instead of the full dictionary.

**4. Retrieve & rerank.** Within that bucket, run hybrid retrieval (dense cosine similarity over the embedding column, fused via reciprocal rank fusion with a lexical match over `long_description`) and rerank the shortlist with a cross-encoder, or Azure AI Search's hybrid query plus semantic ranker if standing up a managed index.

**5. LLM final pick.** GPT-4.1-mini selects from the reranked shortlist, returning structured JSON with the chosen code, a rationale, and a confidence score, explicitly checking the extracted attributes against each candidate's long description.

**6. Confidence gate.** Above a calibrated threshold, auto-accept. Below it, route to the agentic loop rather than guessing.

**7. Agentic Tavily loop.** For sparse or brand/model-only items, an agent (GPT-4.1-mini with a `tavily_search` tool) searches for the actual product, scoped to an approved domain list (manufacturer sites, known distributors, CMS/DME-MAC resources), capped at 2-3 rounds. Enriched attributes feed back into steps 3-5. The agent also runs one targeted search for an explicit code reference; agreement between that finding and the internal pick is treated as corroboration, not as the answer itself.

**8. Escalation to human review.** Items the agent loop can't resolve, or where the web finding conflicts with the internal pick, go to a reviewer with the full context attached (see payload below) so nothing is re-researched from scratch.

**9. Feedback loop.** Every accepted or human-confirmed `(item, supplier, code)` triple writes back into the supplier cache and into the growing dataset used for periodic supervised fine-tuning of GPT-4.1-mini once enough labeled examples accumulate, especially around frequently confused code pairs.

## Key design decisions

Cache before model, always, since it's the cheapest and most accurate path for repeat traffic. Funnel before semantic search, since narrowing the candidate pool by taxonomy removes most of the cross-category confusion that flat retrieval struggles with. Hybrid retrieval, not vector-only, since exact size and unit tokens matter more in HCPCS matching than pure semantic similarity. Web search is evidence, not the verdict — Tavily findings must concur with the internal pick before auto-accepting, and the source is always logged for audit. The 90%+ accuracy target applies to the auto-accepted subset, not to every raw prediction; the confidence gate is what makes that number honest.

## Monitoring

Track auto-accept rate, review queue volume and time-to-resolution, the rate at which auto-accepted codes are later overturned, and how often the Tavily agent's findings agree versus conflict with the internal pick. Drift in any of these is the signal to recalibrate the confidence threshold or refresh the fine-tuning set.

## Next steps

Recompute taxonomy centroids periodically as the master dictionary or confirmed-mapping volume grows. Once a few hundred labeled examples accumulate per frequently-confused code pair, fine-tune GPT-4.1-mini via supervised fine-tuning on Microsoft Foundry. Consider index-time augmentation, generating synthetic "real-world phrasing" variants per code, embedding those alongside the official long description, to close the vocabulary gap between formal CMS language and informal supplier item names.
